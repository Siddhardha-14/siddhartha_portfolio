The Challenge

A problem everyone hits but nobody solves well

You've been there — you get a scanned PDF, a screenshot of a table, or a whiteboard photo. The information you need is right there, but you can't select it, copy it, or edit it. So you retype everything by hand. That's the problem.

And the existing tools? They either cost money, require creating an account, or — most frustratingly — they upload your file to a server. If that file is a medical report, a bank statement, or a confidential contract, that's not acceptable.

😤
The Problem
  • You can't select or copy text that's embedded in an image
  • Online OCR tools upload your files to third-party servers
  • Sensitive documents (medical, legal, financial) get compromised
  • Most tools cost money, require sign-up, or have usage limits
  • Dark-background images (infographics, slides) fail on standard OCR
✅
The Solution
  • Runs the full OCR pipeline inside your browser — nothing leaves your device
  • Tries 3 different image processing algorithms and picks the best result
  • Handles dark, light, noisy, and high-contrast images automatically
  • Completely free — no account, no upload limit, no subscription
  • Outputs a properly formatted Word document with headings, bullets, paragraphs
How It Works

From image to Word in 4 steps

The app runs a 4-stage pipeline entirely inside your browser. Here's exactly what happens when you click "Convert":

1
Image Pre-processing

Three versions of your image are created

The browser's Canvas API generates 3 different processed versions of your image simultaneously: Variant A — grayscale with Otsu's binarisation (best for clean printed text), Variant B — inverts dark backgrounds so white-on-dark text becomes readable, Variant C — high-contrast CSS filter pipeline using GPU acceleration, best for noisy or faded images.

2
3-Variant OCR Engine

Tesseract.js reads all 3 and picks the winner

Tesseract.js (a WebAssembly OCR engine) runs on each of the 3 processed images. It counts the meaningful words in each result and keeps whichever version gave the highest word count. This "best-of-three" approach means the tool self-corrects — even if your image has unusual contrast, one of the variants will work.

3
Text Structuring

Raw OCR text is turned into document structure

Raw Tesseract output is messy — it has pipe characters, duplicate lines, orphan symbols, and no concept of headings vs. paragraphs. A custom post-processor cleans everything: it strips noise, detects all-caps headings, identifies bullet lists by prefix characters, deduplicates repeated lines, and groups content into semantic document elements.

4
DOCX Generation

A properly formatted Word document is assembled

docx.js assembles the structured text into a real Microsoft Word file — with Calibri font, justified paragraphs at 12pt, bold headings sized correctly by level, true bullet list paragraphs, and A4 page margins. The file downloads automatically with a timestamped filename. No conversion, no server — it's built locally and handed directly to your browser's download mechanism.

Key Features

What makes it actually useful

100% Private — Zero Upload
Everything runs in your browser. Your images never touch a server. There's no backend, no cloud, no logs.
Batch Processing
Upload multiple images at once. Each one is processed sequentially with accurate per-image progress tracking.
Smart Dark-Image Support
White text on dark backgrounds is automatically detected and inverted before OCR — something most tools fail at completely.
Real Word Formatting
Output has actual headings, bullet lists, and justified paragraphs — not just a wall of unstyled text dumped in a file.
Cancel Anytime
Built with AbortController — clicking Cancel immediately terminates running OCR workers and frees all memory. No hanging processes.
Works Offline
Once loaded, the tool works with no internet connection. It uses Tesseract.js WASM which runs entirely on the CPU in your browser.
Engineering Challenges

Problems I had to solve from scratch

Challenge 1 — Dark images destroy standard OCR
Most OCR engines assume black text on a white background. Infographics and presentation slides often flip this — white text on dark panels. Standard processing produces empty or random output on these. Solution: Variant B computes the average pixel brightness across the entire image. If it's below 128 (dark image), the image is inverted before binarisation, flipping it to black-on-white for Tesseract to read correctly.
Challenge 2 — No single pre-processing algorithm works universally
A filter that enhances a clean scan can completely destroy a noisy photograph. There's no single correct approach. Solution: Rather than choosing one algorithm, I built three different ones targeting different image types: Otsu thresholding for clean prints, adaptive inversion for dark backgrounds, and GPU-accelerated high-contrast CSS filters for noisy photos. The one returning the most meaningful words wins.
Challenge 3 — Raw OCR output is garbage without post-processing
Tesseract produces strings with pipe characters `|`, duplicated lines, single-character orphans, and no semantic structure. Solution: A custom text post-processor strips pipes and backslashes, filters lines that are only symbols or fewer than 2 characters, detects repeated character patterns like `------`, deduplicates semantically identical lines, and classifies remaining lines as headings, list items, or paragraphs using heuristic rules.
Challenge 4 — Cancelling mid-process without memory leaks
Tesseract.js spawns Web Workers. If the user cancels mid-conversion, those workers can be left running, consuming memory indefinitely. Solution: Each `runTesseract()` call returns both the promise and a `terminate()` handle. An AbortSignal listener immediately calls `terminate()` on the active worker the moment the user cancels — cleanly releasing all memory and stopping all computation instantly.
